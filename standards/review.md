# Code Review 偏好

## 审查维度

按优先级排序：

### 1. 正确性

- 逻辑是否正确
- 边界条件是否处理
- 错误处理是否完整

### 2. 可维护性

- 命名是否清晰（变量、函数、组件）
- 是否有不必要的复杂度
- 是否遵循 DRY（但不过度抽象）

### 3. 性能

- 是否有不必要的重渲染
- 是否有内存泄漏风险
- 大列表是否做了虚拟化

### 4. 一致性

- 同类组件样式是否统一
- 命名风格是否一致
- 间距 / 颜色是否遵循 design token

### 5. 可访问性

- ARIA 属性是否正确
- 键盘导航是否支持
- 对比度是否达标

## UI 审查标准

### 视觉层级

- 信息密度是否合理
- 主操作是否突出

### 色彩系统

- 颜色是否一致
- 对比度是否达到 WCAG AA

### 间距节奏

- padding / gap / margin 是否遵循统一 spacing scale

### 组件一致性

- 同类元素（按钮、卡片、标签）样式是否统一

### 空状态 & 加载态

- 是否有合适的 empty state 和 loading 反馈

### 交互反馈

- hover / focus / disabled 状态是否完整

### 商业感

- 整体是否达到 B2B SaaS 商业软件标准

## 输出格式

对每个维度给出：

- 当前评分（1–10）
- 具体问题（引用文件路径和行号）
- 改进建议（可直接执行的代码级建议）

最后输出优先级排序的 Top 5 改进项。
