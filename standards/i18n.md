# 国际化（i18n）规范

> 仅当系统需要多语言 / 出海 / 多地区部署时启用。

## 核心原则

- i18n 只负责文案替换与格式化，不改变业务行为
- 禁止将业务逻辑写进翻译资源或依赖不同语言产生不同逻辑分支
- i18n key 仅用于替换 UI 文案，不作为业务判断依据

## Key 命名

- 点号分隔：`a.b.c`
- 能反映位置（翻译人员一眼看出属于哪个页面 / 组件 / 区块）
- 全局唯一（全仓库不允许重复 key）
- 同一文案不同位置 → 不同 key（key 是"位置 ID"，不是"语义 ID"）
- 严格禁止复用任何 key（包括 `app.common.*` 通用池）
- 使用位置个数与 key 个数严格 1:1

推荐结构：`app.<module>.<view>.<block>.<name>`

```
✅ app.user.user-list.header.title
✅ app.user.user-list.table.empty
✅ app.user.user-detail.form.save
❌ text1、msg_001、title
```

## 文案硬编码

UI 可见文案禁止硬编码在组件 / 模板中，必须通过 i18n 资源获取：
- 按钮、提示、校验错误、空态、标题
- ElMessage / toast / confirm / dialog 文案
- 枚举映射到 UI 的状态文案

## 占位参数

- 必须用数组传参：`t('key', [arg0, arg1])`
- 禁止对象传参：`t('key', { name })`
- 文案用位置占位：`Welcome, {0}` / `共 {0} 条`

## 拼接转换

仅对 UI 文案生效，识别并转换以下模式：

```
❌ `共${count}条`           → ✅ t('key', [count])，文案：共 {0} 条
❌ '你好' + name            → ✅ t('key', [name])，文案：你好 {0}
❌ {{ '结果：' + val }}     → ✅ {{ t('key', [val]) }}，文案：结果：{0}
```

占位数量与顺序必须严格一致，禁止继续拼字符串。

## 复数与格式化

- 涉及数量变化 → 使用 i18n 方案的复数能力，禁止三元 / 拼接硬凑
- 日期 / 时间 / 数字 / 货币 → 统一格式化入口，禁止散落 `toLocaleString()`

## 资源文件

- 扁平 JSON：`{ "a.b.c": "文案" }`，禁止嵌套对象
- 源语言固定一种（如中文），其他语言从源语言派生
- 新增 key 先写占位值：`"提交-en"` / `"提交-th"`
- 禁止空字符串或缺 key

## 禁止裸 key 传播

禁止在业务代码中传递 / 导出 / 缓存裸 i18n key（如 `const key = 'a.b.c'`）。

```
// ❌ 禁止
const key = 'app.order.pay.status.success'
const text = t(key)

// ✅ 集中使用
const { t } = useI18n()
const statusText = ORDER_PAY_STATUS_TEXT[state.status] ?? ORDER_PAY_STATUS_TEXT.pending

// ✅ 函数映射
const ORDER_PAY_STATUS_TEXT = {
  success: () => t('app.order.pay.status.success'),
  failed: () => t('app.order.pay.status.failed'),
  pending: () => t('app.order.pay.status.pending'),
} as const
```

例外：框架约定只能放 key 的场景（如 `route.meta.titleKey`），允许保存 key，但：
- 仅用于展示，不参与业务判断
- key 仍需按命名规范
- 渲染时统一在一个入口翻译

## 清理

无引用 key 必须及时删除。建议提交前 / CI 扫描：所有 locale 文件的 key 必须至少被代码引用 1 次。

## AI 提取流程

当需要提取页面国际化时，按以下顺序执行：

### 提取范围

- `<template>` 中所有可见纯文本
- 组件 props 默认文案（placeholder / label / title）
- `script setup` 中的 toast / confirm / dialog / 校验规则 message / 兜底错误信息
- 枚举映射到 UI 的状态文案

### 忽略范围

- 测试文件：`**/*.test.*`、`**/*.spec.*`、`**/__tests__/**`
- scripts / mock / playground / demo 目录
- `console.*`、`debugger`
- 非 UI 文案：URL、日志 key、路由 path、className、test id、注释

### 执行顺序

1. 扫描所有字符串字面量与 template 纯文本
2. 过滤非 UI 文案
3. 每个 UI 文案使用位置生成唯一 key（根据文件路径推导）
4. 动态部分做占位转换：`{0}`、`{1}`…
5. 替换为 `t(key)` 或 `t(key, [args])`
6. 输出替换清单 + locale 增量

### 输出格式

```
[替换清单]
- 原文案: ...
  key: ...
  位置: template/script/校验/弹窗/状态映射
  替换为: t('...') 或 t('...', [...])

[locale 增量]
- zh: { "key": "文案" }
- en: { "key": "文案-en" }
只输出新增 key，不输出全量文件。
```
